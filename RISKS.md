# zsk 风险与潜在问题

## 一、功能风险

- **Agent 构建质量依赖模型能力** — `build` 模式的分类和合并由 agent 的 LLM 完成，弱模型可能分类错误、遗漏合并、生成不当摘要
- **标题归一化覆盖不全** — 目前只剥离 `agent/ai` 前缀，其他语言变体、缩写（RAG ↔ Retrieval-Augmented Generation）仍无法匹配
- **Markdown 解析鲁棒性** — 代码块检测仅处理 ` ``` ` 和 `~~~`，嵌套代码块或非常规写法可能漏解析
- **分类体系固定** — 8 个分类硬编码，若研报涉及新领域（如"Agent 部署运维"）无处归类

## 二、性能风险

- **全量 JSON 读写** — 每次修改都重写整个文件，节点上千时性能下降
- **HTML 数据内嵌膨胀** — 节点数多时 HTML 单文件可达几十 MB，浏览器卡顿
- **导入时重复解析** — `build` 模式每次都重新解析所有研报，报告数量大时耗时长

## 三、数据安全

- **无版本控制** — `knowledge_base.json` 修改后无法回滚
- **无并发锁** — 多个进程同时操作可能互相覆盖
- **明文存储** — JSON 和 HTML 中内容未加密，涉密研报需注意分发安全

## 四、兼容性

- **Windows `hermes` 需在 PATH 中** — 否则 `build.bat` 找不到命令
- **Skill 更新不同步** — `skills/` 模板更新后，已部署机器需重新 `python kb.py setup`
- **`import` 可被人类绕过** — 封锁仅针对非交互终端，人类输入 `y` 仍可执行规则模式

## 五、使用风险

- **误删节点不可恢复** — `kb.py delete --cascade` 无回收站
- **setup 需手动执行** — 第一次使用必须记得运行，否则 agent 找不到 skill
- **研报格式差异大** — 无标题层级的纯文本研报需 agent 语义提取，结果不稳定
